This document denes the technical requirements for a college-rst rental marketplace
focused on students, owners, and admins. The system is designed around a practical, stable
architecture using Vite + React for the web frontend, React Native for mobile, FastAPI for
backend APIs, Firebase Authentication for identity, Firestore for primary data storage,
Cloudinary for media, and OpenStreetM ap + Leaet for map display.[cite:29][cite:31][cite:37]
The architecture should optimize for long-term maintainability, simple team onboarding, low
operational overhead, and predictable scaling. The system should avoid unnecessary
microservices, message buses, or heavy infrastructure until actual tra c and complexity
justify them.[cite:40 ][cite:43][cite:47]
The platform should use a modular monolith architecture.
W eb frontend: Vite + React + Tailwind application for student browsing, owner
dashboard, and admin panel.
M obile frontend: React Native app for student and owner usage.
Backend API: FastAPI service exposing REST APIs for business logic, validation,
moderation, search orchestration, and authorization checks.
Authentication: Firebase Authentication for sign-in and user identity tokens.
Database: Firestore as the primary operational database.
M edia storage: Cloudinary for image upload, optimization, and delivery.
M aps: OpenStreetM ap with Leaet for listing and college map display.[cite:29][cite:31]
[cite:35][cite:37]
This system should be deployed as a single backend service with clear internal modules
rather than separate microservices. A layered architecture with API, service, and repository
boundaries is a stable pattern for FastAPI applications and improves testability without
unnecessary complexity.[cite:40 ]

The goal is not to maximize theoretical scale on day one. The goal is to keep the architecture
simple while avoiding future dead ends.
Use modular monolith boundaries so modules can later be separated if truly needed.
[cite:40 ]
Keep services stateless so the API can be horizontally scaled behind a load balancer.
Avoid local lesystem dependency for uploads because Cloudinary already externalizes
media storage.
Use structured logs and health endpoints from the beginning.[cite:40 ]
Design queries around indexed access patterns.[cite:47]
Precompute aggregates such as rating summaries and listing counters.[cite:44]
Avoid large fan-out writes where unnecessary.
Denormalize selectively for high-read listing screens.[cite:44]
Archive stale listings rather than keeping every state in hot queries.
For V1, Firestore-backed ltered listing search is acceptable if lters are kept practical. If
search volume, compound lters, or ranking complexity grows signicantly, a search engine
such as Algolia or OpenSearch can be introduced later. That should be an evolution step, not
a starting dependency.
Use event tracking from the start but keep dashboards limited to high-value operational
metrics. If admin reporting becomes too expensive to compute on demand, add scheduled
aggregation jobs before adding a data warehouse.
The following decisions improve long-term stability without overengineering:
Strong API versioning from day one.
Clear domain modules for users, colleges, properties, reviews, moderation, and analytics.
Centralized auth middleware.
Typed request and response contracts.
Separate approval and visibility states on listings.
Audit logging for admin operations.
